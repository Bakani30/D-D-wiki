🐉 D&D AI Dungeon Master: Core Architecture & Concepts

เอกสารนี้รวบรวมแนวคิดหลักและกรอบการทำงานสำหรับการพัฒนา AI Dungeon Master ที่เน้นความถูกต้องของกฎกติกา ความจำระยะยาว และความสามารถในการขยายสเกล (Scalability)
1. ปรัชญาการออกแบบระบบ (Core Design Philosophy)

    Intent/Effect Separation (การแยกเจตจำนงและผลลัพธ์): AI จะไม่แก้ไขสถานะของเกมโดยตรง (เช่น ไม่ประกาศว่าศัตรูตาย) แต่ AI จะส่ง "เจตจำนง (Intent)" ออกมาให้ Rules Engine ที่เป็นโค้ดโปรแกรม (เช่น โค้ดที่เขียนด้วย Python) ทำการทอยเต๋า คำนวณค่า AC และสรุปผลลัพธ์ (Effect) ก่อนที่จะให้ AI บรรยายภาพเหตุการณ์

    Multi-Model Architecture (การใช้ AI หลายตัวทำงานร่วมกัน): เพื่อประหยัดต้นทุนและเพิ่มประสิทธิภาพ ควรแบ่งหน้าที่ให้ AI แต่ละรุ่น:

        Narrative Agent (รุ่นใหญ่/ฉลาด): เน้นการสวมบทบาท NPC และบรรยายฉาก

        State Inference Agent (รุ่นเล็ก/เร็ว): คอยอ่านบทสนทนาเพื่ออัปเดตสถานะของโลกหลังฉาก (เช่น อัปเดตอารมณ์ของ NPC)

        Relevance Checker (รุ่นเล็ก/เร็ว): คอยตรวจสอบว่าผู้เล่นเดินไปเตะ "เงื่อนไข (Trigger)" ของเหตุการณ์ในอนาคตหรือไม่

    Dice Rolling Engine แยกต่างหาก: การสุ่มตัวเลขทั้งหมดต้องทำผ่านระบบ Code ไม่ใช่ให้ AI จินตนาการตัวเลขขึ้นมาเอง เพื่อป้องกันปัญหา AI ลำเอียงหรือ Hallucination

2. การจัดการความจำและโลกของเกม (World State & Persistence)

นำแนวคิด LLM Wiki Pattern มาประยุกต์ใช้กับ Obsidian เพื่อให้ AI ทำหน้าที่เป็น "ผู้ดูแลจักรวาลเกม (World Keeper)"

    The Wiki as Truth: สถานะปัจจุบันของโลก (HP ของมอนสเตอร์, ไอเทมในกระเป๋า, ความสัมพันธ์กับ NPC) จะถูกเก็บเป็นไฟล์ Markdown (.md) ใน Obsidian

    Structured Story Memory & Importance Decay: บันทึกข้อเท็จจริงในเกมลงฐานข้อมูล เหตุการณ์สำคัญจะคงอยู่ตลอดไป แต่รายละเอียดเล็กๆ จะค่อยๆ เลือนหายไป (Decay) เพื่อประหยัด Token ในระยะยาว

    Consequence Seeds: การฝัง "เมล็ดพันธุ์แห่งผลลัพธ์" เช่น หากผู้เล่นทำร้ายพ่อค้า AI จะบันทึก Trigger ไว้ในระบบ เมื่อเวลาผ่านไป พ่อค้าอาจจ้างนักฆ่ามาล้างแค้น

    Operations (วงจรการทำงาน):

        Ingest: หลังจบ Session ให้ AI สรุปเรื่องราวและวิ่งไปอัปเดตไฟล์ NPC/สถานที่ ที่เกี่ยวข้องทั้งหมด

        Query: ระหว่างเล่น AI จะดึงข้อมูลจากไฟล์อ้างอิงมาตอบโดยไม่มั่วข้อมูล

        Lint: ให้ AI ตรวจสอบความย้อนแย้งของเนื้อเรื่องเป็นระยะ (เช่น NPC ตายแล้วแต่ยังมีชื่อโผล่มาในเควสต์)

3. แผนการพัฒนาและขยายสเกล (Development Roadmap)

การแบ่งระยะการพัฒนาเพื่อก้าวจากการเป็นแค่แชทบอทไปสู่แพลตฟอร์มที่สมบูรณ์ โดยสามารถใช้ Stack อย่าง Python สำหรับระบบหลังบ้าน (Rules/AI Logic) และใช้ React สำหรับทำหน้าจอแสดงผลได้เป็นอย่างดี

    Phase 1: MVP & Core Engine * สร้าง Rules Engine พื้นฐาน (ทอยเต๋า, คิด Damage)

        เชื่อมต่อ API ให้ AI บรรยายผลการต่อสู้

        สร้างหน้าจอ Combat Tracker พื้นฐานเพื่อจัดการ ลำดับเทิร์น (Initiative) และ HP

    Phase 2: Agentic Workflow * แยกระบบ AI ออกเป็นหลายตัว (Narrative, Judge, Inference)

        ให้ระบบสามารถอนุมานการเปลี่ยนแปลงของโลกจากคำพูดผู้เล่นได้อัตโนมัติ

    Phase 3: Persistence & World Building * เชื่อมต่อระบบแชทเข้ากับคลังข้อมูล Obsidian อย่างเต็มรูปแบบ

        นำระบบ Ingest, Query และ Lint มาใช้งานเพื่อบันทึกประวัติศาสตร์แคมเปญ

    Phase 4: Full Scale Web App * สร้าง Web Dashboard เต็มรูปแบบสำหรับผู้เล่นและ AI DM (แสดงแผนที่, กระเป๋าช่องเก็บของ)

        ผูก Database จริง (เช่น MySQL หรือ MongoDB) แทนที่ไฟล์ Markdown เพื่อรองรับการสเกลแบบ Multiplayer
